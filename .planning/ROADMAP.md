# Roadmap: Timur.Shop

## Overview

Действующий интернет-магазин кожаных изделий проходит доработку до production-качества. Путь: сначала закрываем критические уязвимости и обеспечиваем безопасную запись данных (Phase 1), затем строим серверную систему заказов с уведомлениями (Phase 2), добавляем админ-интерфейс для управления заказами (Phase 3), внедряем динамические отзывы с модерацией (Phase 4), и завершаем полировкой верстки и анимаций (Phase 5).

## Phases

**Phase Numbering:**
- Integer phases (1, 2, 3): Planned milestone work
- Decimal phases (2.1, 2.2): Urgent insertions (marked with INSERTED)

Decimal phases appear between their surrounding integers in numeric order.

- [ ] **Phase 2: Orders Data & API** — Серверное сохранение заказов, статусы, Telegram-уведомления
- [ ] **Phase 3: Orders Admin UI** — Админ-панель для просмотра и управления заказами
- [ ] **Phase 4: Dynamic Reviews** — Динамические отзывы с модерацией через API
- [ ] **Phase 5: UI & Polish** — Мобильная адаптивность, анимации, аккуратные компоненты

## Phase Details

### Phase 1: Security Foundation
**Goal**: Магазин безопасен для production — нет критических уязвимостей, данные защищены от повреждения и потерь
**Depends on**: Nothing (first phase)
**Requirements**: SEC-01, SEC-02, SEC-03, SEC-04, SEC-05, SEC-06, SEC-07, SEC-08
**Success Criteria** (what must be TRUE):
  1. Сервер не запускается без переменной окружения ADMIN_PASSWORD — хардкод-пароль удалён
  2. Загрузка SVG-файлов отклоняется сервером, XSS через изображения невозможен
  3. JSON-файлы остаются корректными после принудительного завершения сервера во время записи (атомарная запись)
  4. В папке data/backups/ автоматически создаются резервные копии с вращением
  5. На всех страницах вместо фейковых контактов (+7999000000, href="#") отображаются реальные данные владельца
**Plans:** 2 plans

### Phase 2: Orders Data & API
**Goal**: Покупатель может оформить заказ через форму, заказ сохраняется на сервере и владелец получает уведомление
**Depends on**: Phase 1
**Requirements**: ORD-01, ORD-02, ORD-03, ORD-04, ORD-08
**Success Criteria** (what must be TRUE):
  1. Покупатель заполняет форму заказа на странице корзины, заказ сохраняется в data/orders.json
  2. Корзина очищается только после подтверждения сохранения заказа на сервере
  3. Заказ содержит имя, телефон, адрес/комментарий, способ доставки, товары с ценами, сумму, дату и статус «Новый»
  4. Статусы заказов подчиняются валидной модели переходов (Новый → Подтверждён → В работе → Отправлен → Выполнен → Отменён)
  5. Владелец получает уведомление в Telegram при новом заказе (если бот настроен через env vars)

### Phase 3: Orders Admin UI
**Goal**: Владелец управляет заказами через админ-панель — просмотр, детали, смена статуса
**Depends on**: Phase 2
**Requirements**: ORD-05, ORD-06, ORD-07
**Success Criteria** (what must be TRUE):
  1. Владелец видит список всех заказов на странице admin-orders.html с номером, именем покупателя, суммой, статусом и датой
  2. Владелец открывает детальную страницу заказа и видит состав товаров, контакты покупателя и комментарий
  3. Владелец меняет статус заказа через интерфейс админки, и изменение сохраняется на сервере

### Phase 4: Dynamic Reviews
**Goal**: Покупатели оставляют отзывы, владелец модерирует их, одобренные отзывы отображаются динамически
**Depends on**: Phase 1
**Requirements**: REV-01, REV-02, REV-03, REV-04, REV-05, REV-06
**Success Criteria** (what must be TRUE):
  1. Покупатель оставляет отзыв с текстом, именем и оценкой 1-5 звёзд на странице отзывов
  2. Новые отзывы сохраняются со статусом pending и не видны другим покупателям до одобрения владельцем
  3. Владелец видит очередь отзывов на модерацию в админке и может одобрить или удалить каждый
  4. Одобренные отзывы отображаются на reviews.html динамически через API, включая мигрированные 8 существующих отзывов
  5. На странице отзывов отображается средний рейтинг и общее количество отзывов
